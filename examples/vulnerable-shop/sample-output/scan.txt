╭─  2. 🔴 [CRITICAL] SQL query built by glueing strings together ────────────────────────────────╮
│                                                                                                │
│  lib/db.ts:7                                                                                   │
│  CWE-89  ·  A03:2021-Injection  ·  ATT&CK T1190  ·  confidence: high                           │
│                                                                                                │
│  🎯 Reachable (server-side) from app/api/products/route.ts                                     │
│                                                                                                │
│  SQL query is concatenated with a variable — a user can inject their own SQL. Use a            │
│  parameterized query with ? / $1 placeholders.                                                 │
│                                                                                                │
│  Found here:                                                                                   │
│    return client.query('SELECT * FROM products WHERE name LIKE %' + term + '%');               │
│                                                                                                │
│  💡 Why this matters                                                                           │
│  When you build a SQL query by adding (`+`) a variable into the query                          │
│  string, the database can't tell your query apart from data the user typed.                    │
│  If that variable came from a form, a URL, or an API request, an attacker can                  │
│  type SQL of their own — for example ending your query early and adding                        │
│  `OR 1=1 --` to dump every row, or `; DROP TABLE users; --` to delete data.                    │
│  This is the classic "SQL injection" bug, and it can leak or destroy your                      │
│  entire database.                                                                              │
│                                                                                                │
│  🛠  How to fix it                                                                              │
│  Never put user data directly into the query text. Use a parameterized query                   │
│  (also called a "prepared statement"): write `?` or `$1` placeholders and                      │
│  pass the values as a separate array. The database driver then treats them as                  │
│  pure data that can never become SQL. If you use an ORM (Prisma, Drizzle,                      │
│  Sequelize), use its query builder instead of raw string SQL.                                  │
│    // instead of: db.query('SELECT * FROM users WHERE id = ' + id)                             │
│    db.query('SELECT * FROM users WHERE id = ?', [id]); // value is sent separately             │
│                                                                                                │
│  📚 Learn more                                                                                 │
│     • https://owasp.org/www-community/attacks/SQL_Injection                                    │
│     •                                                                                          │
│  https://cheatsheetseries.owasp.org/cheatsheets/SQL_Injection_Prevention_Cheat_Sheet.html      │
│                                                                                                │

│                                                                                                │
│  app/api/chat/route.ts:14                                                                      │
│  CWE-79  ·  A03:2021-Injection  ·  ATT&CK T1059.007  ·  confidence: medium                     │
│                                                                                                │
│  🎯 Reachable (server-side) from app/api/chat/route.ts                                         │
│                                                                                                │
│  innerHTML assigned a non-literal value; renders as HTML and can execute injected scripts if   │
│  user-controlled.                                                                              │
│                                                                                                │
│  Found here:                                                                                   │
│    document.getElementById('a').innerHTML = answer; // LLM output rendered as HTML (XSS)       │
│                                                                                                │
│  💡 Why this matters                                                                           │
│  Assigning a string to `element.innerHTML` parses it as HTML. When the string contains data    │
│  an attacker controls, they can inject markup that runs JavaScript in your users' browsers     │
│  (cross-site scripting), letting them hijack accounts or steal data.                           │
│                                                                                                │
│  🛠  How to fix it                                                                              │
│  Use `element.textContent` to insert text safely, or build DOM nodes explicitly. In React,     │
│  render `{value}` instead of touching innerHTML. If raw HTML is unavoidable, sanitize with     │
│  DOMPurify first.                                                                              │
│    element.textContent = userInput; // rendered as text, never executed                        │
│                                                                                                │
│  📚 Learn more                                                                                 │
│     • https://owasp.org/www-community/attacks/xss/                                             │
│                                                                                                │
